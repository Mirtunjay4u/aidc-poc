## 2. Objective

This document defines the observability event model for the AIDC Phase 1 backend/API prototype.

The purpose is to make the local demo more operationally credible by defining what should be visible in logs during:

- API startup
- startup validation
- scenario start
- scenario reset
- current scenario read
- hall summary read
- rack records read
- GPU screen read
- unknown scenario request
- API shutdown

This document began as the event model for Phase 1 observability. Structured event logging has now been implemented in api/app_v1.py and validated through runtime smoke testing.

## 4. Observability Principles

The Phase 1 prototype should follow these principles:

1. Every important demo action should leave a readable operational trace.
2. Logs should show which scenario was requested or activated.
3. Startup should clearly show validation success or failure.
4. Unknown scenario requests should be visible and structured.
5. GPU screen reads should be distinguishable from hall/rack reads.
6. Logs should support technical review without implying production monitoring maturity.

## 5. Current Logging Position

The current backend already includes lightweight application logging for startup validation, shutdown, and unexpected failures.

This model extends the intended discipline by defining a consistent event vocabulary for current and future logging improvements.

## 6. Event Taxonomy

The following event names should be used as the standard vocabulary for Phase 1 backend/API observability.

| Event name | Trigger | Purpose |
|---|---|---|
| api_startup_initiated | API application startup begins | Confirms runtime startup has started |
| startup_validation_started | Required file validation begins | Confirms validation gate is running |
| startup_validation_passed | Required file validation completes successfully | Confirms API can safely serve file-backed responses |
| startup_validation_failed | Required file validation fails | Identifies missing or invalid startup dependency |
| api_ready | Application startup completes | Confirms API is ready for local demo use |
| scenario_current_requested | GET /scenario/current is called | Tracks current scenario state read |
| scenario_start_requested | POST /scenario/{scenario_id}/start is called | Tracks requested scenario activation |
| scenario_started | Supported scenario is successfully started | Confirms scenario controller update |
| scenario_reset_requested | POST /scenario/reset is called | Tracks reset action |
| scenario_reset_completed | Scenario state resets to baseline | Confirms reset completion |
| hall_summary_requested | GET /hall/summary/{scenario_id} is called | Tracks hall summary read |
| rack_records_requested | GET /hall/racks/{scenario_id} is called | Tracks rack-level data read |
| gpu_screen_requested | GET /gpu/screen/{scenario_id} is called | Tracks separate GPU screen backend read |
| unknown_scenario_requested | Unsupported scenario ID is requested | Tracks negative-path scenario request |
| api_shutdown_completed | Application shutdown completes | Confirms clean shutdown |

## 7. Standard Event Fields

Every event should follow a consistent field model where practical.

| Field | Required | Description |
|---|---|---|
| event_name | Yes | Standard event name from the taxonomy |
| timestamp | Yes | Event timestamp generated by runtime logging |
| service | Yes | Service identifier, expected aidc-poc-api |
| version | Yes | API version, expected v1 |
| level | Yes | Log severity level |
| scenario_id | Conditional | Scenario ID when the request or action is scenario-specific |
| endpoint | Conditional | API endpoint related to the event |
| method | Conditional | HTTP method related to the event |
| status_code | Conditional | HTTP status code for request/response events |
| result | Conditional | Operational result such as passed, failed, running, reset, or not_found |
| error_code | Conditional | Structured error code for error events |
| message | Yes | Human-readable event message |

## 8. Event Severity Guidance

| Severity | Usage |
|---|---|
| INFO | Normal startup, validation success, supported reads, scenario start/reset, shutdown |
| WARNING | Unknown scenario request or recoverable negative path |
| ERROR | Missing required file, invalid artifact, startup validation failure, unexpected runtime failure |

Phase 1 should avoid noisy logs. The goal is to produce enough evidence for demo review and troubleshooting, not full production observability volume.

## 9. Event Examples

### 9.1 Startup validation passed

```text
event_name=startup_validation_passed
service=aidc-poc-api
version=v1
level=INFO
result=passed
```

## 10. Demo Observability Expectations

During a Phase 1 demo, logs should make the following flow visible:

1. API startup begins.
2. Required file validation starts.
3. Required file validation passes.
4. API becomes ready.
5. Health and scenario list are read.
6. Scenario is started.
7. Hall summary is read.
8. Rack records are read.
9. GPU screen response is read.
10. Scenario is reset.
11. API shuts down cleanly.

The demo operator should be able to explain the runtime flow using the logs without claiming production monitoring maturity.

## 11. Implementation Guidance

Current implementation now includes structured event logging in api/app_v1.py. Future logging improvements should continue to:

- reuse the event names defined in this document
- include scenario_id for scenario-specific events
- include endpoint and method for API request events
- use WARNING for unknown scenario requests
- use ERROR for startup validation failure or unexpected runtime failure
- avoid logging secrets, credentials, tokens, or sensitive infrastructure details
- keep the local demo logs readable and concise

## 12. Validation Guidance

Observability validation should check that the following are visible during runtime:

| Validation item | Expected evidence |
|---|---|
| API startup | api_startup_initiated or equivalent startup log |
| Startup validation | startup_validation_started and startup_validation_passed |
| Scenario start/reset | scenario_start_requested, scenario_started, scenario_reset_completed |
| GPU screen read | gpu_screen_requested |
| Unknown scenario | unknown_scenario_requested with error_code=unknown_scenario |
| Shutdown | api_shutdown_completed |
